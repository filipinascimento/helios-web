# Layout Delegation & Interpolation Guide

**Date:** 2026-01-28

This guide explains how to build layouts that delegate positions, how interpolation fits in, and what buffer patterns to prefer for performance.

## Mental model
- **Sparse attribute buffers** (network attributes) are indexed by node/edge IDs and must be read via index arrays because inactive IDs can exist.
- **Dense buffers** are contiguous repacks of active nodes/edges used by renderers. They are fast for GPU upload but can be expensive to rebuild.
- **Delegates** own position storage independently from the network and can feed positions to layouts and the renderer.

## Position ownership options
### 1) Network-owned (default)
Layouts read/write `_helios_visuals_position` directly on the network.
- Use when layouts update every frame and no external position owner exists.
- Renderer consumes dense buffers generated by helios-network.

### 2) Delegate-owned (CPU)
Use a `PositionDelegate` when you want layouts to write into a separate buffer.
- `CpuMirrorPositionDelegate`: copies from network once, then owns its buffer.
- `ExternalBufferPositionDelegate`: wraps a user-managed buffer.

```js
helios.positions({
  source: 'delegate',
  delegate: new ExternalBufferPositionDelegate(myPositions),
});
```

### 3) Delegate-owned with network interpolation
When interpolation backend is `"network"`, Helios automatically attaches a delegate (if missing) so layouts write into the delegate buffer. The network then interpolates toward that target each frame.

```js
helios.interpolation({
  enabled: true,
  backend: 'network',
  smoothing: 6,
  minDisplacementRatio: 0.0005,
});
```

## Interpolation options
### CPU backend
- Captures dense snapshots and interpolates in JS.
- Best when using dense overrides and no WASM path is available.
- Trades memory for smoothness (two snapshots + interpolated buffer).

### Network backend (C core)
- Captures layout targets into a WASM-backed buffer.
- Each geometry frame advances `_helios_visuals_position` toward the target.
- Avoids per-frame JS copies when the target buffer lives in the WASM heap.
- Avoids per-frame JS copies when the target buffer lives in the WASM heap.
- Target selection order: layout payload positions (if provided), then delegate view when interpolation is configured to use the delegate (default for network backend), otherwise the network position attribute.
- If the network does not expose `interpolateNodeAttribute(...)`, Helios falls back to the CPU backend.

## Buffer safety rules (WASM)
- **Allocate first, view second.** A heap growth can invalidate typed array views.
- Use `withBufferAccess(...)` for any read/write that might follow allocations.
- Avoid caching WASM-backed views across calls that might allocate.

## Performance & copy hotspots
- **Worker layouts** copy positions when sending them to the WebWorker. This happens per layout update, not per render frame.
- **CPU interpolation** duplicates dense buffers for previous/current snapshots.
- **Dense overrides** in `CpuMirrorPositionDelegate.getDenseOverrides()` repack sparse data into contiguous arrays for rendering.
- **Network interpolation** avoids per-step copies when the target is WASM-backed; only layout updates copy into the target.

## When to use dense overrides
- Use dense overrides when the renderer must draw from a delegate-owned buffer.
- Avoid rebuilding dense overrides every frame unless layout changes warrant it.
- Prefer tracking versions/topology to skip redundant repacks.

## Recommended patterns
- **Fast, continuous layouts:** network-owned positions + no interpolation.
- **Sparse layout updates:** delegate-owned positions + network interpolation.
- **External layouts:** delegate that exposes external buffers (planned).

In the basic demo, set the layout update interval to 1–2 seconds to make interpolation visible.
